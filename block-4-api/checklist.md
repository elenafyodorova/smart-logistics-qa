# Чек-лист тестирования контроллера /api/v2/orders

**Источники:**
- Описание контроллера `/api/v2/orders` (таблица методов, URL, статус-кодов)
- Microsoft Graph REST API Guidelines (модель ошибок, поведение методов, правила работы с правами доступа)
- Google AIP / Cloud API Design Guide (общие принципы Get/List/Create/Update/Delete — раздел Standard Methods)

## 1. GET /api/v2/orders — Получение списка заказо

| ID   | Что проверяем                                    | Ожидаемый результат                             |
| ---- | ------------------------------------------------ | ----------------------------------------------- |
| L-01 | В системе есть заказы                            | 200 OK, возвращается список заказов             |
| L-02 | Заказов нет                                      | 200 OK, возвращается пустой список `[]`         |
| L-03 | Есть удалённые заказы                            | 200 OK, удалённые заказы не отображаются        |
| L-04 | Запрос с фильтрацией, сортировкой или пагинацией | 200 OK, данные соответствуют параметрам запроса |
| L-05 | Нет прав на просмотр заказов                     | 403 Forbidden                                   |
| L-06 | Ошибка БД или сервиса                            | 503 Service Unavailable                         |

## 7. Сквозные проверки (применимы к нескольким методам)

### Формат ошибок

Все ответы с ошибками должны соответствовать модели ошибок Microsoft Graph: объект `error` с обязательным полем `code`, человекочитаемым `message` и опциональными `target`/`innererror.code`. Отдельная проверка — что `error.code` в camelCase соответствует именно тому HTTP статусу, который реально вернулся (например, `badRequest` для 400), а не рассинхронизирован с фактическим кодом ответа.

### Права доступа

| ID   | Что проверяем     | Ожидаемый результат          |
| ---- | ----------------- | ---------------------------- |
| A-01 | Запрос без токена | 401 Unauthorized             |
| A-02 | Недостаточно прав | 403 Forbidden                |
| A-03 | Доступ разрешён   | Успешное выполнение операции |


### Превышение лимитов и недоступность сервиса

| ID   | Что проверяем                          | Ожидаемый результат                                                                |
| ---- | -------------------------------------- | ---------------------------------------------------------------------------------- |
| E-01 | Любая ошибка клиента (400, 404 и т.д.) | Возвращается объект `error`                                                        |
| E-02 | Ошибка сервера                         | Сообщение не содержит технических деталей (stack trace, строки подключения и т.п.) |


## 8. Дополнительно

Для практической реализации этого чек-листа имеет смысл:

- Собрать Postman-коллекцию по каждому методу контроллера (List/Get/Create/Update/Delete/Head) с отдельными запросами на happy path и на каждый из негативных сценариев из этого чек-листа.
- Настроить отдельные Postman-окружения (Environment) — как минимум `dev` и `staging` — с переменными `baseUrl` и токеном авторизации, чтобы не дублировать коллекцию под каждый стенд.
- При необходимости добавить автотесты прямо в Postman через вкладку Tests (`pm.test(...)`), которые автоматически проверяют ожидаемый статус-код и структуру тела ответа (особенно формат объекта `error`) — это превращает чек-лист в исполняемый набор проверок, который можно гонять в CI.
