# Анализ бизнес-процесса ЭТрН по BPMN

**Основание для анализа:** BPMN-диаграмма процесса ЭТрН из вложения `etrn-process.png` и комплект XSD-схем из архива `14975031_3xsd.zip`.

## 1. Цель процесса

ЭТрН — это сквозной электронный документ, который создаётся грузоотправителем, передаётся через оператора ЭДО, последовательно обрабатывается перевозчиком, водителем и грузополучателем, а затем закрывается после подтверждения получения груза и фиксации всех необходимых сведений.

Процесс в диаграмме построен как цепочка сообщений и подтверждений между участниками. Для тестирования важно проверять не только заполнение полей, но и корректность переходов между ролями, обработку замечаний, изменения реквизитов и завершение документа.

## 2. Участники процесса и их роль

В BPMN выделяются следующие дорожки:

| Участник | Роль в процессе |
|---|---|
| Грузоотправитель | Создаёт ЭТрН, заполняет обязательные сведения, инициирует отправку, при необходимости вносит изменения и подтверждает действия |
| Оператор ЭДО | Принимает, маршрутизирует и доставляет документ между участниками документооборота |
| Перевозчик | Получает ЭТрН, проверяет данные, подтверждает или отклоняет сведения, фиксирует замечания, отправляет документ дальше |
| Водитель | Участвует в операциях, связанных с погрузкой/выгрузкой, передаёт данные диспетчеру и подтверждает выполнение действий |
| Грузополучатель | Получает ЭТрН, проверяет груз, подтверждает приём либо фиксирует замечания |

## 3. Сквозной бизнес-процесс (happy path)

Ниже — обобщённая последовательность, которая прослеживается по BPMN.

1. **Грузоотправитель создаёт ЭТрН.**  
   Старт процесса начинается с создания документа.

2. **Грузоотправитель заполняет обязательные сведения о грузе и перевозке.**  
   На этом этапе документ должен пройти первичную проверку обязательных полей и форматов.

3. **Документ сохраняется в черновик.**  
   Пользователь может сохранить заполнение до отправки.

4. **Грузоотправитель подписывает и отправляет ЭТрН.**  
   Далее документ передаётся в контур ЭДО.

5. **Оператор ЭДО принимает документ и маршрутизирует его адресату.**  
   Это технический шаг доставки между участниками.

6. **Перевозчик получает ЭТрН и подтверждает получение.**

7. **Перевозчик проверяет сведения по перевозке.**  
   Если всё корректно, документ продвигается дальше; если есть расхождения — фиксируются замечания.

8. **При наличии замечаний перевозчик вносит несоответствия и отправляет документ на согласование/корректировку.**  
   Ветвление по BPMN показывает, что замечания не завершают процесс, а переводят его в корректирующий сценарий.

9. **Водитель участвует в операциях погрузки/выгрузки и передаёт данные диспетчеру.**  
   Это важно для подтверждения факта перевозки.

10. **Грузополучатель получает ЭТрН, проверяет груз и подтверждает приём.**  
    При расхождениях также возможна фиксация замечаний.

11. **Документ закрывается после подтверждения выдачи груза и завершения всех обязательных действий.**

## 4. Обобщённая модель статусов для тестирования

В самой диаграмме акцент сделан на событиях передачи и подтверждения, поэтому для тестирования удобно использовать обобщённую модель статусов:

- **Черновик**
- **Подготовлен к отправке**
- **Отправлен оператору ЭДО**
- **Получен перевозчиком**
- **Проверен перевозчиком**
- **Передан на согласование/корректировку** — если есть замечания
- **Получен грузополучателем**
- **Подтверждён приём груза**
- **Завершён**

Для каждого перехода важно проверять:
- доступность следующего шага только после завершения предыдущего;
- неизменность уже подписанных/зафиксированных реквизитов;
- корректное отображение текущего статуса всем участникам;
- корректную обработку отказов и ошибок доставки.

## 5. Ветвления и исключения, которые видны в BPMN

### 5.1. Замечания по документу или грузу
Если у перевозчика или грузополучателя есть замечания, процесс не должен завершаться “успешно” по основному сценарию.  
Ожидаемая реакция системы:
- фиксация замечаний;
- перевод в сценарий корректировки;
- сохранение истории изменений;
- отправка документа по альтернативной ветке согласования.

### 5.2. Изменение сведений о грузополучателе и/или порте назначения
В диаграмме предусмотрен отдельный сценарий изменения реквизитов.  
Ожидаемое поведение:
- инициатор изменения создаёт событие изменения;
- перевозчик подтверждает или отклоняет изменение;
- участники видят актуальные сведения без потери истории.

### 5.3. Подтверждение фактов погрузки/выгрузки
Водитель и перевозчик участвуют в фиксации этапов перевозки.  
Тестировать нужно:
- доступность действий только в нужный момент;
- правильность передачи данных диспетчеру;
- сохранение отметок о выполненных операциях.

### 5.4. Завершение перевозки
После подтверждения получения груза и передачи всех обязательных сведений документ должен перейти в финальный статус.  
Не должно быть возможности вернуться к редактированию закрытых частей без отдельного сценария исправления.

## 6. Что критично проверить по процессу

1. Создание документа с первого шага и сохранение черновика.
2. Отправка только после выполнения обязательных проверок.
3. Корректная маршрутизация между участниками через оператора ЭДО.
4. Обработка замечаний и корректировок без потери истории.
5. Корректное отображение реквизитов у всех ролей.
6. Смена статусов только по допустимым переходам BPMN.
7. Поведение системы при ошибках связи, подписи, неполных данных и повторной отправке.

## 7. Вывод

BPMN показывает не один “линейный” документ, а цепочку согласованных действий между несколькими ролями.  
Для качественного тестирования ЭТрН нужно проверять:
- создание и первичное заполнение;
- межролевое движение документа;
- сценарии замечаний и исправлений;
- подтверждение погрузки/выгрузки;
- финальное закрытие документа;
- устойчивость к ошибкам передачи и валидации.
