# Чек-лист / тест-кейсы по ЭТрН

**Источники:**
- BPMN-диаграмма процесса работы с ЭТрН (`etrn-process.png`)
- XSD-схема Титула 1 «Информация грузоотправителя»: `ON_TRNVPRGO_1_973_21_05_01_01.xsd` (Приложение № 3 к Приказу ФНС России от 14.05.2024 № ЕД-7-26/383@)

## 1. Основной бизнес-процесс ЭТрН

Процесс на диаграмме построен вокруг последовательного оформления четырёх титулов документа разными участниками. Тест-кейсы сгруппированы по титулам, чтобы было видно, какой участник на каком шаге отвечает за какие данные и какую развилку проходит.

### 1.1. Титул 1 — формирует грузоотправитель

| ID    | Проверка                                                                                                                                    | Ожидаемый результат                                                                                                           |
|-------|---------------------------------------------------------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------------------------------------------------|
| T1-01 | Создать ЭТрН, заполнить обязательные сведения первого титула (условия договора, стороны сделки, водитель, груз, сопроводительные документы) | Документ создаётся, обязательные поля принимаются без ошибок                                                                  |
| T1-02 | Сохранить заполненные данные как черновик                                                                                                   | Черновик сохраняется, данные не теряются при повторном открытии                                                               |
| T1-03 | Подписать и отправить первый титул                                                                                                          | Документ уходит оператору ЭДО, статус ЭТрН меняется на «Ожидает отгрузки со склада» (дорожка «Грузоотправитель» на диаграмме) |
| T1-04 | Попытаться отправить титул 1 с незаполненными обязательными полями                                                                          | Отправка блокируется, отображается список незаполненных обязательных полей                                                    |
| T1-05 | Оператор ЭДО получает первый титул и маршрутизирует его                                                                                     | Документ доходит до перевозчика без искажения реквизитов (дорожка «Оператор ЭДО»)                                             |

### 1.2. Титул 2 — формирует перевозчик при погрузке

| ID    | Проверка                                                                                     | Ожидаемый результат                                                                                                   |
|-------|----------------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------------------------------------------|
| T2-01 | Водитель передаёт диспетчеру данные о погрузке                                               | Данные становятся доступны перевозчику (дорожка «Водитель» → «Перевозчик»)                                            |
| T2-02 | Перевозчик подтверждает получение первого титула, проверяет данные, замечаний нет            | Развилка «Есть замечания?» проходится по ветке «Нет», перевозчик подписывает титул 2                                  |
| T2-03 | Перевозчик активирует чекбокс «Замечания при погрузке» и вносит данные в поля несоответствий | Развилка проходится по ветке «Да»; поля несоответствий становятся обязательными для заполнения именно в этом сценарии |
| T2-04 | Подписать и отправить титул 2                                                                | Статус ЭТрН меняется на «Груз принят к перевозке», водителю генерируется и отправляется QR-код                        |
| T2-05 | Оператор ЭДО рассылает второй титул грузоотправителю и грузополучателю                       | Обе стороны получают актуальный титул 2 без расхождений с тем, что подписал перевозчик                                |

### 1.3. Титул 3 — формирует грузополучатель при выгрузке

| ID    | Проверка                                                                                          | Ожидаемый результат                                                                                                  |
|-------|---------------------------------------------------------------------------------------------------|----------------------------------------------------------------------------------------------------------------------|
| T3-01 | Водитель прибывает на место разгрузки, сообщает диспетчеру                                        | Грузополучатель видит, что транспорт ожидается / прибыл                                                              |
| T3-02 | Грузополучатель принимает транспорт на разгрузку, проходит развилку «Принять груз?» по ветке «Да» | Активируется чекбокс «Груз принят», заполняются данные о приёмке, титул 3 подписывается и отправляется               |
| T3-03 | Грузополучатель проходит развилку «Принять груз?» по ветке «Нет»                                  | Процесс уходит в отдельную финальную ветку «Отказ от груза», документ не продолжает движение по основному happy path |
| T3-04 | Подписанный титул 3 доходит до грузоотправителя и перевозчика через оператора ЭДО                 | Стороны видят подтверждение приёмки груза                                                                            |

### 1.4. Титул 4 — формирует перевозчик по итогам разгрузки

| ID    | Проверка                                                                                                      | Ожидаемый результат                                                                                 |
|-------|---------------------------------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------------------------|
| T4-01 | Перевозчик ожидает завершения разгрузки, получает сведения о выгрузке, замечаний нет                          | Развилка «Обнаружены замечания по выгрузке?» проходится по ветке «Нет», титул 4 подписывается       |
| T4-02 | Перевозчик активирует чекбокс «Замечания при выгрузке» и вносит данные в поля несоответствий                  | Развилка проходится по ветке «Да»; поля несоответствий обязательны                                  |
| T4-03 | Оператор ЭДО рассылает четвёртый титул грузоотправителю и грузополучателю и направляет третий титул в ГИС ЭПД | Финальный статус ЭТрН — «Груз принят грузополучателем / Завершён»; процесс закрывается как успешный |

### 1.5. Сквозные проверки по процессу

| ID    | Проверка                                                                                                                                       | Ожидаемый результат                                                                                                                                                                                                                                                                              |
|-------|------------------------------------------------------------------------------------------------------------------------------------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| CR-01 | На каждом переходе между титулами проверить, что статус документа меняется именно в момент подписания, а не раньше                             | Статус не должен «забегать вперёд» относительно реально выполненного действия участника                                                                                                                                                                                                          |
| CR-02 | Проверить ветку с замечаниями (Титул 2 и Титул 4): возможно ли поставить чекбокс «Есть замечания» без заполнения ни одного поля несоответствий | Ожидается блокировка — чекбокс и поля несоответствий должны быть синхронизированы                                                                                                                                                                                                                |
| CR-03 | Изменение грузополучателя и/или порта назначения после подписания титула 1                                                                     | **Требует уточнения**: явная ветка такого изменения не видна на предоставленной BPMN-диаграмме. В XSD-схеме Титула 1 присутствует поле `ИспрТрН` (исправление ЭТрН) — возможно, изменение реализуется через него. Создан [Issue](https://github.com/elenafyodorova/smart-logistics-qa/issues/1). |

## 2. Ключевые проверки формы (документа по XSD)

### Ошибки и пользовательская реакция

| ID     | Сценарий                                                                                                   | Ожидаемое поведение                                                                        |
|--------|------------------------------------------------------------------------------------------------------------|--------------------------------------------------------------------------------------------|
| ERR-01 | Имя переданного файла не совпадает со значением атрибута `ИдФайл`                                          | Отклонение с понятным сообщением об ошибке, а не тихий пропуск несоответствия              |
| ERR-03 | У участника сделки (`СвГО`/`СвГП`/`СвПрв`) не указан ни один из четырёх допустимых вариантов идентификации | Отклонение с указанием, какой участник и какое поле не заполнено                           |
| ERR-04 | `ВесГруз` указан отрицательным или равным нулю                                                             | Ожидается отклонение бизнес-логикой (XSD-схема сама по себе это не блокирует — см. п. 2.2) |
| ERR-05 | `СпосПодтПолном = 3`, но `СвДоверЭл` не передан                                                            | Отклонение с явным указанием на недостающую доверенность                                   |

## 3. Минимальный набор граничных проверок

| Проверка                                         | Входные данные                                         | Почему критично                                                                                                                                                                 |
|--------------------------------------------------|--------------------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Граница времени                                  | `23:59:59` (валидно), `24:00:00` (невалидно)           | Стандартная проверка верхней границы суток                                                                                                                                      |
| Знак числовых полей груза                        | `ВесГруз = -1`, `ВесГруз = 0`                          | Схема не содержит `minInclusive`, поэтому это зона ответственности бизнес-логики, а не XSD                                                                                      |
| Длина ИНН                                        | 9 и 11 цифр для `ИННЮЛТип` (норма — 10)                | Проверка, что отклонение длины именно отклоняется, а не обрезается/дополняется системой                                                                                         |
